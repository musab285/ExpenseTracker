# Deployment Guide - Railway

This guide covers deploying the ExpenseTracker backend to Railway.

## Prerequisites
- Railway account (railway.app)
- PostgreSQL database (Railway provides this)
- Git repository

## Deployment Steps

### 1. Connect Your Repository to Railway
1. Go to [railway.app](https://railway.app)
2. Create new project → Import Git Repo
3. Select this repository
4. Railway will automatically detect the Django app

### 2. Configure Environment Variables

In Railway dashboard, add the following environment variables:

```
DEBUG=False
SECRET_KEY=<generate-a-secure-key>
ALLOWED_HOSTS=<your-railway-domain>.railway.app

# Database variables (Railway auto-provides these, but ensure they're set)
DATABASE=<your_db_name>
user=postgres
PASSWORD=<your_db_password>
HOST=<postgres_host>
PORT=5432

# Frontend CORS (update with your frontend domain)
CORS_ALLOWED_ORIGINS=https://<your-frontend>.railway.app,https://yourdomain.com
```

### 3. Generate SECRET_KEY

Run locally to generate a secure key:
```bash
python -c "from django.core.management.utils import get_random_secret_key; print(get_random_secret_key())"
```

### 4. Add PostgreSQL Plugin

1. In Railway dashboard → New
2. Add Database → PostgreSQL
3. Railway automatically injects DATABASE_URL and POSTGRES_* variables
4. Update your environment variables with the connection details

### 5. Deploy

- Push to main/master branch - Railway auto-deploys
- Or manually trigger deployment in Railway dashboard

### 6. Verify Deployment

```bash
# Check logs in Railway dashboard
# Run migrations via Railway CLI:
railway run python manage.py migrate

# Create superuser:
railway run python manage.py createsuperuser

# Test endpoints:
curl https://<your-app>.railway.app/api/profiles/
```

## Key Files for Deployment

- **Procfile** - Defines how Railway runs the app (migrations + gunicorn)
- **runtime.txt** - Specifies Python 3.13.0
- **requirements.txt** - All Python dependencies with pinned versions
- **settings.py** - Updated for production:
  - DEBUG from environment variable
  - SECRET_KEY from environment variable
  - ALLOWED_HOSTS configurable
  - Static files handled by WhiteNoise
  - CORS origins configurable

## Production Checklist

✅ DEBUG = False
✅ SECRET_KEY is unique and secure
✅ ALLOWED_HOSTS includes your domain
✅ Database is PostgreSQL (not SQLite)
✅ CORS_ALLOWED_ORIGINS set for frontend
✅ Static files served via WhiteNoise
✅ Migrations run automatically (via Procfile release step)
✅ Environment variables from .env.example are set
✅ Gunicorn serves the app (4 workers, 9 threads each)

## Troubleshooting

**Migrations not running:**
```bash
railway run python manage.py migrate
```

**Static files not loading:**
- Ensure STATIC_ROOT is set (it is in settings.py)
- WhiteNoise middleware is installed

**Database connection errors:**
- Verify DATABASE, user, PASSWORD, HOST, PORT are correct
- Ensure PostgreSQL plugin is connected
- Check SSL mode (sslmode=require in settings)

**CORS errors:**
- Update CORS_ALLOWED_ORIGINS in environment variables
- Must be HTTPS in production

## Manual Commands via Railway CLI

```bash
# Install Railway CLI (one-time)
npm i -g @railway/cli

# Login
railway login

# Run migrations
railway run python manage.py migrate

# Create superuser
railway run python manage.py createsuperuser

# Check logs
railway logs

# Open shell
railway shell
```

## Production Architecture

```
Frontend (React) 
  ↓ (HTTPS)
Railway App (Django + Gunicorn)
  ↓
PostgreSQL Database (Railway)
```

## Security Notes

- All traffic is HTTPS
- SECRET_KEY is securely stored as Railway environment variable
- Database password is stored as Railway environment variable
- Never commit .env to git (use .env.example as template)
- Use strong SECRET_KEY generated by Django
